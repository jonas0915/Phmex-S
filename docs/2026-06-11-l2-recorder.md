# L2 Tick Recorder (Tier 1) — 2026-06-11

Standalone daemon recording tick-level order-book data for offline whale-detection
research (the precondition named in docs/2026-06-01-imbalance-reversion-edge-findings.md).
Zero coupling to the live bot — separate process, public WebSocket, no API keys.

- Script: `scripts/l2_tick_recorder.py` · launchd: `com.phmex.l2-recorder`
  (KeepAlive daemon) · log: `logs/l2_recorder.log`
- Symbols: BTC, ETH, INJ, ARB (USDT perps). Depth: top 5 levels/side
  (depth 10 measured ~1.05 GB/day on 6/11 — over budget; halved).
- Output: `logs/l2_ticks/<SYMBOL>/<UTC-date>.jsonl` — one line per book update:
  `{ts: local recv ms, et: exchange ts ms, n: book nonce, sym, b:[[px,sz]×5],
  a:[[px,sz]×5]}`. `ts − et` measures our latency per update (89 ms on first
  sampled BTC line); the nonce detects feed gaps offline. Gzip-rotated daily,
  deleted after 14 days, hard-paused if the dir exceeds 5 GB (resumes < 90% cap).
- Excluded from the iCloud backup (reproducible research data).
- Transport: ccxt.pro `watch_order_book` (ccxt 4.4.100, `watchOrderBook: True`
  for phemex) — same lib as ws_feed.py; it maintains the book from Phemex
  incremental messages and handles ping/pong + reconnect. Trade-off: unified
  API doesn't expose which side/level changed, so full top-5 book is recorded
  per update; diff consecutive lines to reconstruct deltas.
- Measured 6/11 ~8:07–8:12 PM PT, 0 parse errors both runs, clean SIGTERM exits:
  - Test 1, depth 10, 113.9 s, 3,449 updates: BTC 15.2/s, ETH 11.0, INJ 2.2,
    ARB 2.0 → 12,186 B/s ≈ **1,053 MB/day** → over budget, depth halved.
  - Test 2, depth 5, 87.9 s, 2,473 updates: BTC 15.4/s, ETH 9.1, INJ 2.0,
    ARB 1.7 → 6,901 B/s ≈ **569 MB/day**. gzip on real data: 16.3× →
    archived day ≈ 35 MB; steady state ≈ 1 GB, well under the 5 GB cap.
  - Caveat: quiet evening sample — daytime rates will run higher; 60 s stats
    lines in `logs/l2_recorder.log` show live rates; disk guard is the backstop.
  - Phemex quirk: update cadence is throttled server-side and scales with
    activity (BTC ~15/s vs INJ/ARB ~2/s, consistent across both runs).

Purpose gate: after ~1-2 weeks of data, test offline whether whale accumulation is
detectable at our latency BEFORE any event-driven (tier 3) rebuild is considered.
